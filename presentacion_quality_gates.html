Fundamento del Proyecto

¿Qué son los Quality Gates?

Un Quality Gate es una barrera de calidad automática dentro de un pipeline CI/CD. Su función es bloquear el avance del código si este no cumple con los estándares mínimos definidos por el equipo.

El Problema que Resuelven

❌ Sin Quality Gates ✅ Con Quality Gates
Código defectuoso llega a producciónSolo código de calidad avanza
Los bugs se detectan tarde (en producción)Los bugs se detectan en el commit
Revisión de código manual y subjetivaValidación automática y objetiva
Deuda técnica acumulada constantementeCalidad mantenida de forma continua
Estándares inconsistentes entre desarrolladoresMismo estándar para todo el equipo

Rol de ngrok

Como Jenkins corre localmente, GitHub no puede enviarle webhooks directamente. ngrok genera una URL pública temporal que apunta a nuestro puerto 8080. Cuando hacemos git push, GitHub notifica a ngrok, y este redirige la señal a Jenkins para disparar el pipeline automáticamente.

# Iniciar ngrok en el puerto de Jenkins ngrok http 8080 # URL generada (ejemplo): https://a1b2c3.ngrok.io → localhost:8080

¿Qué mide JaCoCo?

JaCoCo instrumenta el bytecode de Java para detectar qué líneas de código fueron ejecutadas durante los tests y cuáles nunca se tocaron. El resultado es un porcentaje llamado "cobertura".

Configuración en pom.xml

<!-- JaCoCo: falla el build si la cobertura es menor al umbral --> <plugin> <groupId>org.jacoco</groupId> <artifactId>jacoco-maven-plugin</artifactId> <rules> <limit> <counter>LINE</counter> <minimum>0.80</minimum> <!-- mínimo 80% --> </limit> <limit> <counter>BRANCH</counter> <minimum>0.70</minimum> <!-- mínimo 70% --> </limit> </rules> </plugin>

Quality Gate #3

Análisis Estático: Checkstyle

Checkstyle analiza el estilo y estructura del código sin ejecutarlo. Compara el código fuente contra un conjunto de reglas basadas en el Google Java Style Guide. Cualquier violación detiene el pipeline.

Reglas Configuradas (checkstyle.xml)

📛
Naming Conventions: Clases en PascalCase, métodos en camelCase, constantes en UPPER_CASE. Los nombres deben ser descriptivos.
📦
Imports: Sin imports no usados, sin import *, sin imports duplicados o redundantes.
🏗️
Estructura: Llaves {} obligatorias en todos los bloques. Bloques vacíos prohibidos sin comentario.
📏
Complejidad: Máx. complejidad ciclomática: 10. Máx. líneas por método: 50. Máx. por archivo: 500.
🔤
Espaciado: Espacios alrededor de operadores, sin tabulaciones (solo espacios), sin espacios en blanco al final de línea.
Ejemplo: Código que FALLA
// ❌ Import no utilizado import java.util.Date; // ❌ Bloque sin llaves if (x > 0) doSomething(); // ❌ Variable mal nombrada int A = 5;
Ejemplo: Código que PASA
// ✅ Solo imports necesarios import java.util.List; // ✅ Siempre con llaves if (x > 0) { doSomething(); } // ✅ Nombre descriptivo int totalCount = 5;
# Resultado exitoso en el pipeline: [INFO] Starting audit... [INFO] Audit done. → 0 violations found [INFO] BUILD SUCCESS
